# 国标麻将联机验收清单

这份清单用于在真实服务器环境下验证 MahjongPaper 的国标麻将流程是否可用、可继续对局、且不会在第二局或复杂反应场景中失稳。

建议至少准备 4 个可控座位，优先使用 4 名真人玩家；如果人数不够，可以混用 Bot，但吃、碰、杠、抢杠和、多家响应优先级等场景最好仍由真人配合完成。

## 测试前准备

- [ ] 服务端已经正常加载 `MahjongPaper` 与 `CraftEngine`
- [ ] 客户端资源包已生效，能正确看到麻将牌正反面、花牌和桌面模型
- [ ] 测试人员具备以下权限：`/mahjong create`、`/mahjong rule`、`/mahjong state`、`/mahjong forceend`
- [ ] 记录本次测试的插件版本、服务端核心版本、世界名、参与玩家名单
- [ ] 如需验证排行或持久化，提前确认数据库配置可用

## 开桌与切换为 GB

- [ ] 执行 `/mahjong create` 创建一张牌桌
- [ ] 执行 `/mahjong list` 记录 `tableId`
- [ ] 4 名玩家分别通过 `/mahjong join <tableId>` 或点击座位标签入座
- [ ] 执行 `/mahjong rule variant GB`
- [ ] 可选执行 `/mahjong state`，确认当前规则摘要已经切到 GB
- [ ] 4 名玩家分别执行 `/mahjong start`

预期结果：

- [ ] 牌局成功开始
- [ ] 庄家起手可打 14 张，闲家在轮到自己前保持正确张数
- [ ] 桌面中央状态摘要显示的是 GB 规则信息，而不是日麻专用信息
- [ ] 不应出现立直、宝牌、里宝牌等仅属于 Riichi 的提示

## 起手、花牌与补花

- [ ] 至少验证一次起手带花
- [ ] 至少验证一次摸牌后补到花
- [ ] 确认花牌不会错误留在手牌可打区域，而是显示到公开花牌区域
- [ ] 确认补花后玩家手牌张数恢复正确
- [ ] 如果补花后再次摸到花牌，确认会继续补花直到拿到可操作的普通牌或进入流局

预期结果：

- [ ] 花牌公开展示正确
- [ ] 补花不会把花牌残留在可打手牌里
- [ ] 后续和牌结算能反映花牌带来的番种或番数

## 吃、碰、杠动作

### 吃

- [ ] 构造一个只能由下家吃牌的场景
- [ ] 下家执行 `/mahjong chii <tileA> <tileB>`
- [ ] 确认被吃的弃牌从牌河中移除
- [ ] 确认吃牌后由吃牌者继续出牌，不会额外摸一张普通牌

### 碰

- [ ] 构造一个可碰的场景
- [ ] 执行 `/mahjong pon`
- [ ] 确认被碰的弃牌从牌河中移除
- [ ] 确认碰牌后由碰牌者继续出牌，不会额外摸一张普通牌

### 明杠

- [ ] 构造一个可明杠的场景
- [ ] 执行 `/mahjong minkan`
- [ ] 确认被杠的弃牌从牌河中移除
- [ ] 确认明杠后会从牌墙尾部补牌
- [ ] 如果补牌是花牌，确认会继续补花

### 暗杠与加杠

- [ ] 构造一个可暗杠的场景，执行 `/mahjong kan <tile>`
- [ ] 确认暗杠后会从牌墙尾部补牌
- [ ] 构造一个碰后可加杠的场景，执行 `/mahjong kan <tile>`
- [ ] 确认加杠后同样会从牌墙尾部补牌

## 抢杠和与多人响应优先级

- [ ] 构造一个玩家加杠时，其他玩家可和牌的场景
- [ ] 其他玩家执行 `/mahjong ron`
- [ ] 确认抢杠和优先于加杠成功结算

- [ ] 再构造一张弃牌同时可被多家响应的场景
- [ ] 验证优先级顺序为 `ron > minkan > pon > chii`
- [ ] 未选择动作的玩家执行 `/mahjong skip`

预期结果：

- [ ] 多人响应按优先级正确结算
- [ ] 未被采用的低优先级动作不会错误生效

## 和牌与 8 番门槛

- [ ] 先尝试一个明显不足 8 番的 `/mahjong tsumo` 或 `/mahjong ron`
- [ ] 确认命令失败，且对局不会被错误结束
- [ ] 再完成一个合法的 8 番或以上和牌
- [ ] 自摸时执行 `/mahjong tsumo`，荣和时执行 `/mahjong ron`

预期结果：

- [ ] 不足 8 番时不能和牌
- [ ] 满足条件时可以正常和牌并结算
- [ ] 结算 UI 能显示总番数和番种列表
- [ ] 如果本局含花牌，结算结果里能反映花牌相关内容

## 结算、继续下一局与稳定性

- [ ] 和牌或流局后，确认 `/mahjong settlement` 可以重新打开最近一次结算界面
- [ ] 确认一局结束后玩家需要重新 `/mahjong start` 才会开始下一局
- [ ] 执行 `/mahjong state`，确认轮次、庄位、剩余状态摘要刷新正常
- [ ] 连续至少完成 2 局，确认不会出现“第一局正常、第二局开始失败”的问题
- [ ] 可选执行 `/mahjong forceend <tableId>`，确认管理员强制结束后桌面能回到等待状态

## 建议记录项

如测试中发现异常，建议至少记录以下信息：

- 使用的命令
- `tableId`
- 参与座位与玩家
- 出问题前最后一张摸牌、弃牌或花牌
- `/mahjong state` 输出
- 服务端报错堆栈
- 对应截图或短视频

## 最小通过标准

- [ ] 能从 `/mahjong rule variant GB` 正常开局到完整结算
- [ ] 花牌与补花至少验证通过 1 次
- [ ] 吃、碰、明杠、暗杠、加杠、自摸、荣和至少各成功 1 次
- [ ] 不足 8 番不能和，满足条件可以和
- [ ] 第二局能够正常开始并继续进行
